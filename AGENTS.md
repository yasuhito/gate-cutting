# AGENTS.md

このリポジトリで作業する際のルールです。

## 基本方針

- 依頼を受けたら、可能な範囲で自律的に作業を進める。
- 変更したら、関連する README / docs も更新する。
- 研究コードの原本性をなるべく保ち、整理・共通化・文書化を優先する。

## ドキュメント整理ルール

- 原稿・スライドなどの資料は `docs/` に置く。
- 原稿の実験コードは `experiments/exp1/` と `experiments/exp2/` を正とする。
- `legacy/ex2/` は試作・参考コードとして扱い、必要以上に書き換えない。

## 計画ドキュメントのルール

- **計画書・作業計画・リファクタリング計画は HTML 形式で作る。**
- 新しい計画を追加する場合は `docs/*.html` として保存する。
- 計画用途の `*.md` は追加しない。既存計画を更新する場合も HTML を直接更新する。
- 計画HTMLを追加・リネームしたら、必ず `README.md` の資料一覧も更新する。
- 通常の実装メモや作者コメント整理は Markdown のままでもよいが、「計画」として使う文書は HTML に統一する。

## Stim リファクタリング方針

- Stim 周辺の共通処理は、実験スクリプト内にコピペせず共通モジュールへ切り出す。
- 優先的に共通化する処理:
  - `device.json` からの `ErrorParams` 抽出
  - `qiskit_to_stim()`
  - `append_operation_with_noise()`
  - ノイズ付き Stim 回路生成
  - 測定追加と読み出しエラー挿入
  - Gate Cutting 実行処理
- CX 後に `I` を入れる暫定実装は、可能なら `TICK` など意図が明確な区切りに置き換える。
- 未対応ゲートを黙って無視しない。原則として例外または明示的な警告にする。
- 切断対象は `(control, target)` のペアだけでなく、可能なら命令番号でも管理する。

## 実装方針: TDD

- 実装は **TDD（Test-Driven Development）** で進める。
- 新機能・リファクタリング・バグ修正では、原則として先に失敗するテストを書く。
- テストで期待仕様を固定してから、最小限の実装で通す。
- テストが通った後にリファクタリングする。
- 既存の研究コードを整理する場合も、振る舞いを変えないことを確認する現状仕様テストを先に追加する。
- 大きな実験や長時間ベンチマークではなく、小さな簡易テスト / 単体テストを優先して TDD サイクルを短く保つ。

## テスト設計ルール: アサーション 1 つ原則

- 各テストケースのアサーションは **1 つ** にする。
- テスト名で、そのテストケースが何を検証しているかを明確にする。
- 複数のアサーションが必要に見える場合は、複数のテストケースに分割する。
- 複数の性質をまとめて検証するために、巨大な dict/list の一括比較へ逃げない。対象の振る舞いごとにテストを分ける。
- 例外を検証する場合の `assertRaises` は、そのテストケースの唯一のアサーションとして扱う。

## レビュールール

- ひとまとまりの作業を終えたら、コミット前に必ず現在の差分へ並列レビューをかける。コードだけでなく `*.html` / `*.md` / `AGENTS.md` などのドキュメント更新も対象にする。
- コード変更のレビュー観点: correctness（正しさ）/ test coverage（テスト網羅性）/ unnecessary complexity（不要な複雑さ）。
- ドキュメント変更のレビュー観点: correctness（記述と実装の整合、リンク切れ、構文崩れ、用語の意味的誤り）/ clarity（想定読者への読みやすさ、見出し階層と情報構造の整合、不明瞭な指示語や未定義略語、ルー語の混入）/ unnecessary complexity（冗長表現、重複説明、空白ノイズ、本筋と無関係な情報）。
- レビュー所見は重大度別に要約し、適用価値のある修正だけを反映する。

## ドキュメントと言語

- README、docs、計画HTML、実装メモ、レビューコメント、作業ログは日本語で書く。
- 日本語文の中では、不自然な英日混在の「ルー語」を避ける。
- 英語の普通名詞に自然な日本語訳がある場合は、日本語または定着したカタカナ語を使う。
  - 例: `old script compatibility wrapper` ではなく「既存実験互換のラッパー」または互換性が不要なら単に「回路生成関数」。
  - 例: `test coverage が不足` ではなく「テスト網羅性が不足」。
  - 例: `fallback を使う` ではなく「代替処理を使う」。
  - 例: `smoke test` ではなく「簡易テスト」。
  - 例: `source layout` ではなく「src 配置」。
- ただし、ツール名、ファイル名、API 名、CLI 引数、環境変数、コード識別子、固有の技術用語（Stim / Qiskit / SciPy / MIP など）は原文のまま書いてよい。

## 実行・検証ルール

- Python ファイルを変更した場合は、最低限 `python -m py_compile ...` などで構文確認する。
- 実験スクリプトは相対パス・同一ディレクトリ import に依存しているため、実行時は対象ディレクトリへ移動して確認する。
- 大きなベンチマークは時間がかかるため、必要に応じて試行回数 / ショット数を小さくした簡易テストを行う。
